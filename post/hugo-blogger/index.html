How Hugo turned me into a blogger wannabe

What is this?

I have decided to finally start a blog. This is just a welcome message (mostly realized to test Hugo, Github Pages and a few themes). In this blog I will talk about the project I work on in my free time; topics range from machine/deep learning to social network analysis, distributed computing and more. Some projects are my own ideas, others are the result of me playing with coll stuff you find around the web. This first article is about the technical details behind the blog itself, what technologies I’m using, how I’ve set up my workflow and how hugo convinced me to start this.

About this article

This is supposed to be a technical blog, a safe shelter for computer scientists and alikes. How I am developing this blog can be interesting and usefull for other people who want to share their projects and their knowledge. The idea here is to give you an overview of how I am developing and updating the blog from a technical perspective.

Resources

Hugo, alongside Github Pages, are the main reasons why I have finally decided to start this project.

Hugo is “a fast and modern static website engine”. That is, a HUGE help when you want to just write your thoughs, without worrying about themes, styles, and general web-development-related issues. It is easy to install and configure (short tutorial is down the road), light, fast (gotta love go), open source… I could go on for miles. There are downsides and negative aspects too, but the moment you realize what Hugo is meant for you also realize that negative aspects turn into features; you will realize this in a few weeks of usage.

Github Pages allows you to create and publish websites for your projects, as well as personal blogs or portfolios. The code for you site is hosted directly from your GitHub repository. Just edit, push, and your changes are live.

The way Hugo and Github Pages interact is mesmerizing; this instruments team up for a produtictivity boost comparable to the introductions of rounded wheels over squared ones. Damn, you gotta love this things; and to bring you on board with me, let’s dive into a tutorial on how to setup this workflow.

Please note that I will assume that you already use git. In case you don’t, here you are (and you’re a bad person).

Let’s start the party

Step 0: Install and configure Hugo

You only need to follow this steps the first time you use Hugo. Since it is written in go, it supports multiple platforms and comes in pre-built binaries available here.

A complete installation guide is available here.

If you’re on a Mac, I advice you install Homebrew and just brew install hugo.

Before you start

If you haven’t already done that, check that Hugo is correctly configured by running

hugo help

You should see something like

hugo is the main command, used to build your Hugo site.

Hugo is a Fast and Flexible Static Site Generator
built with love by spf13 and friends in Go.

Complete documentation is available at http://gohugo.io/.

Usage:
  hugo [flags]
  hugo [command]
  ...

It goes on for a while.

Step 1: Create your first Hugo project

In this short example we will create a blog.

Hugo has a large set of commands to manage your website. To create a new website move to a convenient location and run

hugo new site blog

Change directory to blog and you will see that Hugo has created directories to store all the files required for your website.

drwxr-xr-x  2 user   68 Nov 30 23:58 archetypes
-rw-r--r--  1 user   97 Dec  1 00:22 config.toml
drwxr-xr-x  3 user  102 Nov 30 23:59 content
drwxr-xr-x  2 user   68 Nov 30 23:58 data
drwxr-xr-x  2 user   68 Nov 30 23:58 layouts
drwxr-xr-x  2 user   68 Nov 30 23:58 static
drwxr-xr-x  6 user  204 Dec  1 00:20 themes

The directory has 6 sub-directories and 1 file. Let’s look at each one of them:

  • archetypes: You can create new content files in Hugo using the hugo new command. When you run that command, it adds few configuration properties to the post like date and title. Archetype allows you to define your own configuration properties that will be added to the post front matter whenever hugo new command is used.

  • config.toml: Every website should have a configuration file at the root. By default, the configuration file uses TOML format but you can also use YAML or JSON formats as well. TOML is minimal configuration file format that’s easy to read due to obvious semantics. The configuration settings mentioned in the config.toml are applied to the full site. These configuration settings include baseurl and title of the website.

  • content: This is where you will store content of the website. Inside content, you will create sub-directories for different sections. Let’s suppose your website has three actions – blog, article, and tutorial then you will have three different directories for each of them inside the content directory. The name of the section i.e. blog, article, or tutorial will be used by Hugo to apply a specific layout applicable to that section.

  • data: This directory is used to store configuration files that can be used by Hugo when generating your website. You can write these files in YAML, JSON, or TOML format.

  • layouts: The content inside this directory is used to specify how your content will be converted into the static website.

  • static: This directory is used to store all the static content that your website will need like images, CSS, JavaScript or other static content.

Step 2: Create contents

Let’s add a new post to our blog. We will use the hugo new command to create a new post. In our example, we want to create a welcome post to greet users visiting our blog. Make sure you are inside the blog directory and type

hugo new post/welcome

This will create a new directory post inside the content directory, and a new file named welcome.md inside it.

The content of welcome.md will look like this:

+++
draft = true
title = "welcome"
date = "2016-12-01T00:22:59+01:00"

+++

The content inside +++ is the TOML configuration for the post. This configuration is called front matter. It enables you to define post configuration along with its content. By default, each post will have the three configuration properties shown above.

Let’s add a short welcome message:

+++
draft = true
title = "welcome"
date = "2016-12-01T00:22:59+01:00"

+++

Welcome to my blog!

Step 3: Serve content

Hugo has an inbuilt server that can serve your website content so that you can preview it. You can also use the inbuilt Hugo server in production. Place yourself in the blog directory and run

hugo server

The output will read

0 of 1 draft rendered
0 future content
0 pages created
0 paginator pages created
0 tags created
0 categories created
in 9 ms

You can view your blog at http://localhost:1313, go check it out… it’s a blank page. That’s because:

  1. As you can see in the hugo server command output, Hugo didn’t render the draft. Hugo will only render drafts if you pass the buildDrafts flag to the hugo server command.

  2. We have to specify a theme that Hugo can use to render the content. We will do that in the next step.

To render drafts, run hugo server --buildDrafts and head back to http://localhost:1313. Yet another wonderful blank page. We are missing the theme, remember that?

Step 4: Add theme

Themes provide the layout and templates that will be used by Hugo to render your website. There are a lot of open-source themes available here that you can use.

In order to add a theme move to the themes directory inside the root of your website and clone a theme for your website. For our example, we will use the bleak theme.

Inside the themes directory run git clone https://github.com/Zenithar/hugo-theme-bleak.git

Get back to the blog directory (cd ..) and run hugo server --theme=hugo-theme-bleak --buildDrafts.

Go to http://localhost:1313 and enjoy your freshly crafted blog.

Now let’s get back to code and try to understand how themes work, in order to customize and modify them. A theme consist of the following:

  • theme.toml is the theme configuration file that gives information about the theme like name and description of theme, author details, and theme license.

  • images directory contains four images, screenshot.png, tn.png, full_blog.png and full_post.png. These images are examples of what the theme looks like; as an example screenshot.png is the image of the list view and tn.png is the single post view.

  • layouts directory contains different views for different content types. Every content type should have two files single.html and list.html. single.html is used for rendering a single piece of content. list.html is used to view a list of content items.

  • static directory stores all the static assets used by the template.

Obviously you can clone multiple themes and test theme using the --theme argument.

Step 5: Update config file

The website uses the dummy values specified in blog/config.toml. Let’s update the configuration.

baseurl = "https://mywebsite.com"
title = "My blog"
languageCode = "en-us"

Optional: Add comments

To integrate comments in your blog you can use Disqus. Just open the blog/config.toml file and add

[Params]
  disqusShortname = <your disqus shortname>

Now, commenting will be enabled in your blog.

Step 6: Make post public

Our welcome post is still in draft status. To make a draft public, you can either run this command

hugo undraft content/post/welcome.md

or manually change the draft status in the post to false.

Now you can start the server without the buildDrafts argument.

hugo server --theme=hugo-theme-bleak

Step 7: Generate website

To generate Hugo website source you can use to deploy your website on GitHub pages, first edit blog/config.toml, changing the baseurl line to:

baseurl = "https://username.github.io/"

Then type

hugo --theme=hugo-theme-bleak

After this command, a public folder will be created, containing teh generated source.

Step 8: Go public

On the github side of the thing, create a new empty (no README) repo named username.github.io.

Now move to the public directory and add version control.

git init
git remote add origin https://github.com/username/username.github.io.git
git add --all
git commit -m "First commit"
git push origin master

Visit https://username.github.io/ and enjoy your blog, now available online.

Anytime, you can regenerate the site with:

hugo --theme=hugo-theme-bleak
cd public
git add --all
git commit -m "<some change message>"
git push origin master
